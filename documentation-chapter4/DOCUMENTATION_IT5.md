# Iteration 5 — Cross-Strategy Answer Agreement Gate (Training-Free Gate 1)

> **Design Science Research Artifact:** Replace the trained Gate 1 classifier
> entirely with a training-free heuristic: if the no-retrieval (nor_qa) and
> single-step retrieval (oner_qa) QA strategies produce the same answer for a
> given question, route to A (no retrieval); otherwise, delegate to Clf2's B/C
> prediction. No model is trained, no checkpoint is loaded, no GPU is required
> for the Gate 1 decision.

## 2. Gate 1 Replacement

### 2.1 What is replaced

In IT1–IT4, Gate 1 is a trained T5-Large binary classifier (Clf1) that reads a question and predicts A or R. The classifier requires:
- A fine-tuned model checkpoint (~770 M parameters)
- GPU inference via `transformers` + `torch`
- A training pipeline (`run_classifier.py`) with loss function, optimizer, scheduler

### 2.2 What replaces it

IT5 replaces the entire trained Clf1 with a single Python script that:
1. Loads **pre-existing** QA prediction files (nor_qa and oner_qa — already generated during retrieval experiments)
2. Compares the two answers after normalization
3. If they agree → route to A (no retrieval needed)
4. If they disagree → pass to Clf2 to decide B (single-step) vs C (multi-step)

### 2.3 No trained model is loaded

The script does **not** import `torch`, `transformers`, `accelerate`, or any ML library. There is no `model.load_pretrained()`, no `model.generate()`, no checkpoint path, no GPU allocation. The entire computation is string comparison on pre-computed outputs.

### 2.4 Conceptual motivation

The intuition: if a question can be answered correctly without any retrieval (nor_qa), and single-step retrieval (oner_qa) produces the same answer, then retrieval is unnecessary — the model already "knows" the answer. Disagreement suggests the question is harder and requires retrieval, so Clf2 decides which retrieval strategy to use.

## 3. Uncertainty Estimation Implementation

### 3.1 Overview

The "uncertainty estimation" in IT5 is a degenerate case: it uses exactly **two** pre-computed QA outputs (nor_qa and oner_qa) and checks for exact agreement after normalization. There is no sampling, no temperature variation, no ensemble, and no probabilistic score.

### 3.2 Number of "samples"

Exactly **2**:
1. `nor_qa` — no-retrieval QA (the model answers from parametric knowledge only)
2. `oner_qa` — single-step BM25 retrieval QA (the model answers with one round of retrieved context)

These are defined in `load_strategy_predictions()` [predict_complexity_agreement.py L63–78].

### 3.3 Sampling strategy: NONE — greedy decoding

Both nor_qa and oner_qa predictions were generated by the LLM server with **greedy decoding**:

| Setting | Value | File | Line |
|---|---|---|---|
| `do_sample` | `False` (default) | `commaqa/models/llm_client_generator.py` | [L17, L77, L109, L147] |
| `do_sample` | `False` (server-side default) | `llm_server/serve.py` | [L161] |
| `temperature` | `1.0` (default, but irrelevant when `do_sample=False`) | `commaqa/models/llm_client_generator.py` | [L18] |
| `output_scores` | `False` (hardcoded) | `llm_server/serve.py` | [L200] |

When `do_sample=False`, HuggingFace's `model.generate()` uses **argmax (greedy) decoding** — it always selects the highest-probability token at each step. This means:
- Given the same prompt, the output is **deterministic**
- There is no stochastic variation between runs
- The two "samples" (nor_qa and oner_qa) differ only because their **inputs** differ (with vs without retrieved context), not because of any randomness in generation

### 3.4 Aggregation logic: `compute_agreement()`

The `compute_agreement()` function [predict_complexity_agreement.py L80–109] implements the full comparison pipeline. Here is the step-by-step algorithm:

| Step | Code | Line(s) | Description |
|---|---|---|---|
| 1. Iterate over question IDs | `for qid in qids:` | [L85] | Processes every question in predict.json |
| 2. Load raw answers | `nor_raw = nor_preds[qid]` / `oner_raw = oner_preds[qid]` | [L86–87] | Look up QA prediction by question ID |
| 3. Handle list answers | `if isinstance(nor_raw, list): nor_raw = nor_raw[0] if nor_raw else ""` | [L89–92] | Some prediction files store answers as lists; take first element or empty string |
| 4. Cast to string | `nor_raw = str(nor_raw)` / `oner_raw = str(oner_raw)` | [L93–94] | Ensure string type for downstream processing |
| 5. Extract answer from CoT | `nor_extracted = answer_extractor(nor_raw)` | [L96–97] | Strip chain-of-thought reasoning via regex `".* answer is:? (.*)\.?"` [L53]; if no match, return raw string [L58] |
| 6. Normalize | `nor_norm = normalize_answer(nor_extracted)` | [L99–100] | Lower-case → remove punctuation → remove articles (a/an/the) → collapse whitespace [L34–46] |
| 7. Compare | `agree = bool(nor_norm and oner_norm and nor_norm == oner_norm)` | [L103] | **Exact string match** after normalization. Both must be non-empty. |
| 8. Store result | `results[qid] = {"agree": agree, "nor_answer": nor_raw, "oner_answer": oner_raw}` | [L104–108] | Preserve raw answers for debugging |

### 3.5 `answer_extractor()` details

Defined at [predict_complexity_agreement.py L48–60]:

1. Strip surrounding double quotes [L50–51]
2. Attempt regex match: `".* answer is:? (.*)\\.?"` [L52–53]
3. If match: extract capture group 1, strip trailing period [L54–57]
4. If no match: return the input unchanged [L58–59]

This is the **same** `answer_extractor()` logic used in `evaluate_final_acc.py` [L48–64].

### 3.6 `normalize_answer()` details

Defined at [predict_complexity_agreement.py L34–46]:

1. `lower(s)` — convert to lowercase [L43–44]
2. `remove_punc(text)` — remove all `string.punctuation` characters [L40–42]
3. `remove_articles(text)` — regex-remove "a", "an", "the" as whole words [L36–38]
4. `white_space_fix(text)` — collapse whitespace to single spaces [L38–39]

Applied in order: lower → remove_punc → remove_articles → white_space_fix (via function composition in `return` at [L46]).

This is the **same** `normalize_answer()` logic used in `evaluate_final_acc.py` [L30–47].

### 3.7 Routing logic in `main()`

After `compute_agreement()` returns, the routing decision is at [predict_complexity_agreement.py L133–142]:

```python
for qid in all_qids:
    if agreement[qid]["agree"]:
        merged[qid] = {"prediction": "A", "dataset_name": qid_to_dataset[qid]}
    else:
        merged[qid] = {"prediction": clf2[qid]["prediction"], "dataset_name": qid_to_dataset[qid]}
```

- **Agree → A:** The question is answerable without retrieval.
- **Disagree → Clf2's prediction (B or C):** The question needs retrieval; Clf2 decides which kind.

## 4. Decision Threshold

**There is NO tunable threshold.** The decision is a hard binary: exact string match after normalization.

The critical line is [predict_complexity_agreement.py L103]:

```python
agree = bool(nor_norm and oner_norm and nor_norm == oner_norm)
```

Three conditions must all hold:
1. `nor_norm` is non-empty (truthy)
2. `oner_norm` is non-empty (truthy)
3. `nor_norm == oner_norm` — exact string equality

There is no soft similarity score, no cosine threshold, no F1 overlap, no edit distance tolerance. A single differing character after normalization causes disagreement.

## 6. Data Pipeline

### 6.1 How queries are fed

Questions are **not** fed through a model. Instead, the script loads pre-computed QA prediction files that were generated during prior retrieval experiments (the same files used to evaluate nor_qa, oner_qa, and ircot_qa strategies).

The flow:

```
predict.json ──────────────────────────── qid → dataset_name mapping
                                          (3000 questions across 6 datasets)

predictions/test/nor_qa_{model}_{ds}/     ─── qid → answer (no retrieval)
predictions/test/oner_qa_{model}_{ds}/    ─── qid → answer (single-step retrieval)
                                               ↓
                              compute_agreement(nor, oner, qids)
                                               ↓
                                    agree? ── Yes → A
                                           └─ No  → clf2[qid]["prediction"] (B or C)
                                               ↓
predictions/test/{strategy}_{model}_{ds}/ ─── look up final answer from
                                              the selected strategy's predictions
```

### 6.2 Prediction file format

All prediction files (`prediction__*.json`) are JSON objects mapping question ID → answer string:

```json
{
    "single_nq_dev_9": "Gal Gadot",
    "single_nq_dev_33": "England",
    ...
}
```

500 questions per dataset × 6 datasets = 3 000 total questions.

### 6.3 predict.json format

`classifier/data/musique_hotpot_wiki2_nq_tqa_sqd/predict.json` is a JSON list of objects:

```json
[
    {"id": "2hop__511176_22458", "dataset_name": "musique", "question": "...", "answer": "...", "total_answer": "..."},
    ...
]
```

The script extracts only `id` and `dataset_name` [predict_complexity_agreement.py L117–118]:
```python
qid_to_dataset = {item["id"]: item["dataset_name"] for item in predict_data}
```

### 6.4 BM25 retrieval counts per model

| Model | `ONER_BM25` (oner_qa) | `IRCOT_BM25` (ircot_qa) | Source |

## 10. Suspicious / Noteworthy Items

### 10.1 Not true sampling-based uncertainty estimation

| Issue | Detail |
|---|---|
| **What** | The script is described as "cross-strategy answer agreement" but in the broader UE literature, agreement-based methods typically require **stochastic** sampling (e.g., multiple forward passes with dropout, temperature sampling, or ensemble members). IT5 uses exactly 2 **deterministic** outputs that differ only in input context (with vs without retrieval). |
| **Implication** | This is closer to a **consistency check** between two fixed strategies than true uncertainty estimation. It measures whether retrieval changes the answer, not whether the model is uncertain. |
| **Impact on validity** | The method conflates "retrieval doesn't change the answer" with "retrieval is unnecessary." These are different claims — retrieval might not change a wrong answer. |

### 10.2 `output_scores=False` confirmed — no probability information available

| Issue | Detail |
|---|---|
| **What** | The LLM server hardcodes `output_scores=False` [llm_server/serve.py L200], with the comment `"make it configurable later"`. This means the generated token probabilities are discarded. |
| **Implication** | Even if one wanted to use model confidence (e.g., sequence probability) as a soft agreement signal, the information is not available in the prediction files. The agreement script cannot access generation scores. |
| **File** | [llm_server/serve.py L200] |

### 10.3 No temperature pathology

| Issue | Detail |
|---|---|
| **What** | `temperature=1.0` (default) and `do_sample=False` (default). When `do_sample=False`, temperature has no effect on generation — HuggingFace ignores temperature during greedy decoding. |
| **Confirmation** | [commaqa/models/llm_client_generator.py L17–18]: `do_sample=False, temperature=1.0` defaults. The `LLMClientGenerator.__init__` stores these as instance attributes [L147, L180]. |
| **Impact** | No risk of temperature-induced randomness or overconfident/underconfident distributions affecting the predictions. |

### 10.4 Binary threshold with no soft margin

| Issue | Detail |
|---|---|
| **What** | The agreement decision is a hard binary: exact string match after normalization [predict_complexity_agreement.py L103]. There is no partial-match tolerance — e.g., "Barack Obama" vs "Obama" would be a disagreement after normalization ("barack obama" ≠ "obama"). |
| **Risk** | Synonyms, abbreviations, and minor wording differences cause false disagreements, potentially routing questions to retrieval unnecessarily. This biases toward over-retrieval (more B/C, fewer A). |
| **Mitigation** | The `normalize_answer()` function removes articles, punctuation, and extra whitespace, which handles some trivial differences. But it does not handle semantic equivalence. |

### 10.5 GPT uses different BM25 counts

| Issue | Detail |
|---|---|
| **What** | GPT uses `ONER_BM25=6` and `IRCOT_BM25=3`, while XL/XXL use `ONER_BM25=15` and `IRCOT_BM25=6` [predict_complexity_agreement.py L29–30]. |
| **Implication** | The oner_qa predictions for GPT are based on fewer retrieved documents than for XL/XXL. This means the "signal" from retrieval is weaker for GPT — fewer documents means less chance of finding relevant context, potentially reducing the difference between nor_qa and oner_qa answers. This could inflate agreement rates for GPT relative to XL/XXL. |
| **Origin** | These counts are inherited from the original Adaptive-RAG experimental setup and are consistent with IT1's `predict_complexity_split_classifiers.py`. |

### 10.6 `routing_stats.json` saved — full audit trail

| Issue | Detail |
|---|---|
| **What** | IT5 saves a `routing_stats.json` [predict_complexity_agreement.py L227–241] that records model name, Clf2 source file, method description, agreement rate, A/B/C distribution, total retrieval steps, and per-dataset breakdowns. |
| **Contrast with IT1–IT4** | The trained-classifier routing script (`predict_complexity_split_classifiers.py`) does **not** save equivalent metadata. IT5 is more auditable. |
| **Benefit** | Makes results reproducible and inspectable without re-running the script. |

### 10.7 No premature termination risks

| Issue | Detail |
|---|---|
| **What** | The script has no `try/except` blocks, no error suppression, no `|| echo "[WARN]"` shell fallbacks. If any prediction file is missing or a question ID is not found, the script raises a standard Python `KeyError` or `FileNotFoundError` and terminates immediately. |
| **Implication** | This is a **desirable property** — silent failures are impossible. Any data pipeline issue is surfaced immediately. Contrast with IT4's GPT shell script which tolerates failures via `|| echo "[WARN]"`. |

### 10.8 stepNum loading has a fallback path

| Issue | Detail |
|---|---|
| **What** | The script attempts to load a consolidated `stepNum.json` from `predictions/test/ircot_qa_{model}/total/stepNum.json` [predict_complexity_agreement.py L148–150]. If this file does not exist, it falls back to loading per-dataset `stepNum.json` files [L152–158]. |
| **Risk** | If neither the consolidated nor the per-dataset files exist, `total_step_num` remains an empty dict. Questions routed to C would get `stepNum=0` via `.get(qid, 0)` [predict_complexity_agreement.py L172], which is technically incorrect (C should have stepNum ≥ 1). However, stepNum is only used for cost accounting, not for answer selection. |

### 10.9 Prediction files are loaded redundantly per dataset

| Issue | Detail |
|---|---|
| **What** | In the per-dataset loop [predict_complexity_agreement.py L167–199], `load_json(dataName_to_files[data_name][option])` is called once per question (inside the inner loop over `merged.items()`). This re-reads the same JSON file from disk for every question in that dataset. |
| **Performance** | For 500 questions per dataset, this means 500 `load_json()` calls per strategy per dataset, each re-reading and re-parsing the same file. With 6 datasets and up to 3 strategies, this could be up to ~9 000 redundant file reads. |
| **Impact** | Correctness is not affected — the output is identical. Runtime is slower than necessary but acceptable for 3 000 questions with small JSON files. |
| **Contrast with IT1** | The IT1 routing script (`predict_complexity_split_classifiers.py`) has the same pattern in `save_prediction_with_classified_label()` [postprocess_utils.py L30], where `load_json()` is also called per-question inside the loop. |

### 10.10 The `answer_extractor()` and `normalize_answer()` are duplicated

| Issue | Detail |
|---|---|
| **What** | Both functions are copy-pasted from `evaluate_final_acc.py` into `predict_complexity_agreement.py`. They are not imported from a shared module. |
| **Risk** | If one copy is updated and the other is not, the normalization applied during routing (IT5) and evaluation could diverge, leading to inconsistent results. |
| **Current state** | As of this iteration, both copies are identical. |
